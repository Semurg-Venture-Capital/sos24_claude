## 1. Что это за проект

**Бренд:** **SOS24** · **Домен:** **sos24.uz** *(утверждено 2026-05-05)*
**Регион:** Республика Узбекистан
**Позиционирование:** «помощник 24/7 на дороге» (название обязывает)
**Тип:** мобильное приложение (клиенты) + веб (лендинг + админка + B2B-кабинет партнёров) + бэкенд

### Продукты, которые продаёт платформа
- **ОСАГО** — обязательное страхование автогражданской ответственности
- **КАСКО** — добровольное страхование авто от ущерба/угона
- **Урегулирование убытков** + вызов аварийного комиссара 24/7
- **Партнёрская сеть:** СТО, медклиники (через B2B-кабинет)
- **Расширения в будущем:** «Зелёная карта», страхование пассажиров, доп. услуги

### Бизнес-модель
- На старте — **одна страховая компания** (заказчик)
- В перспективе — архитектура должна позволять подключение **других страховщиков** (готовим под маркетплейс, но не реализуем сейчас)
- **B2B-партнёрская часть:** отдельный кабинет для СТО и медклиник — каталог услуг, заявки от клиентов, финансы, рейтинг
- Есть **B2C-админка** для страховой (модерация, выплаты, отчёты, тарифы, контент лендинга, диспетчер комиссаров)

## 4. Стек (зафиксировано после раунда 1 ответов клиента)

- **Mobile:** React Native (iOS + Android, единая кодовая база)
- **Web (лендинг + админка + B2B-кабинет партнёров):** Next.js 14
- **Backend:** строим **с нуля** на NestJS + PostgreSQL + Redis + MinIO
- **Хостинг:** **self-hosted** на серверах заказчика (без облаков типа Vercel / Heroku / managed-БД)
- **Платежи (приоритет):** **Uzcard прямой эквайринг** (свой биллинг). Click / Payme / Uzum — опционально, по запросу клиента (под вопросом, см. `QUESTIONS2.md` Q-N.3.1)
- **Языки UI:** **uz-Latn, uz-Cyrl, ru, en** — 4 локали, i18n с самого начала
- **Идентификация:** SMS-OTP (Playmobile) + MyID + OneID на старте; e-IMZO — фаза 2 (по остаточному принципу)
- **Госданные:** через **NAPP** (единый шлюз для авто, ВУ, реестра полисов)
- **Бухгалтерия:** интеграция с **1С**
- **SMS:** Playmobile · **Аналитика:** PostHog · **Crash:** Sentry

**Бренд-токены (фиксировано):**

- Шрифты: **Mont** (заголовки/числа), **Inter** (body/UI)
- Логотип: `assets/brand/logo.svg` (красный знак-«S» + чёрный wordmark «SOS24»)
- Референсы: `assets/brand/ref1.jpg` / `ref2.jpg` / `ref3.jpg` (две темы, hero-фото авто, KPI/donut, floating bar). Lime-акцент референсов → у нас красный.

**Мои обязанности по дизайну:**

- Обновлять токены бренда, если дизайнер их уточнит

**Что даёт дизайнер обратно:**

- Макеты экранов из Stitch + ручная полировка

## 6. Правила работы (для меня)

### Что делать
- Любой новый вопрос к клиенту → сразу в `QUESTIONS.md` с контекстом и вариантами
- Любое архитектурное решение → фиксировать в `PLAN.md` с обоснованием
- Любое изменение списка экранов/флоу → обновлять `STITCH.md`
- Сначала уточнять, потом писать код
- Все даты в документах — **абсолютные** (не «через неделю», а «2026-05-11»)

### Чего не делать
- НЕ начинать код, пока не утверждён дизайн (Этап 1 ещё не открыт)
- НЕ выбирать визуальные решения за дизайнера
- НЕ выдумывать ответы на бизнес-вопросы — в `QUESTIONS.md`
- НЕ создавать дублирующие документы — обновлять существующие

## 9. Точка входа в работу

Когда пользователь возвращается:
1. Читаю `CLAUDE.md` (этот файл)
2. Читаю `QUESTIONS.md` — какие вопросы ещё открыты
3. Читаю `PLAN.md` (раздел 0) — что уже зафиксировано, чтобы не переспрашивать
