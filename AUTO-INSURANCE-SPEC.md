## 1. О продукте

Мобильное приложение страховой компании для продажи **автостраховых полисов** физлицам в Узбекистане и управления ими.

**Ценность для клиента:** оформить ОСАГО/КАСКО за несколько минут прямо в телефоне, хранить электронные полисы с QR-проверкой, вести «гараж» автомобилей, оплачивать картой и из кошелька, видеть историю.

**Ценность для страховой:** прямой цифровой канал продаж, автоматизация оформления, снижение бумажной работы.

## 2. Границы первой версии (MVP)

### Входит
- **Авторизация:** SMS-OTP + биометрия/PIN, верификация личности через MyID.
- **Продукты:** оформление **ОСАГО** и **КАСКО** (каталог → расчёт → оформление → оплата → электронный полис с QR).
- **Гараж:** список авто, добавление по техпаспорту (данные из НАПП), деталь, редактирование.
- **Полисы:** список, деталь, электронный полис с QR для проверки, статусы.
- **Профиль:** личные данные (из MyID, read-only), документы (паспорт, водительское удостоверение).
- **Финансы:** кошелёк (баланс/пополнение), привязанные карты, история платежей.
- **Уведомления:** push (напоминания об истечении полиса, статусы оплаты).
- **Локализация:** 4 локали.

### Не входит в MVP (задел на будущее)
- Админ-панель страховой (отдельный этап).
- B2B-кабинет партнёров/СТО.
- Другие виды страхования (здоровье, имущество, жизнь, путешествия).
- Урегулирование убытков / европротокол / аварийный комиссар.
- Маркетплейс нескольких страховщиков.

> Архитектуру закладываем **расширяемой** под перечисленное (типы продуктов, роли), но в MVP реализуем только авто.

## 5. Дизайн-система «Liquid Glass»

> Цветовую схему и макеты предоставляет **дизайнер заказчика**. Ниже — **структура токенов** и **набор компонентов**, которые реализуются независимо от палитры. Стиль «Liquid Glass» (матовое стекло / полупрозрачные слои с блюром) — обязателен и сохраняется.

### 5.1. Что такое Liquid Glass здесь
Полупрозрачные «стеклянные» поверхности: размытие фона (`expo-blur` `BlurView`), полупрозрачная заливка, тонкая светлая обводка (hairline), мягкая тень. Карточки, панели, боттом-шиты, плавающие бары — на стекле. Обязательны **светлая и тёмная** темы.

### 5.2. Токены (структура; значения цветов — от дизайнера)
```ts
// packages/i18n не про это — токены в apps/mobile/src/theme/tokens.ts
export const tokens = {
  color: {
    brand:      '#______',  // основной акцент (даёт дизайнер)
    brandInk:   '#______',  // тёмный бренд
    bg:         '#______',  // фон
    surface:    'rgba(255,255,255,0.55)', // стеклянная поверхность (light)
    hairline:   'rgba(20,20,20,0.08)',    // тонкая обводка
    ink:        '#111111', inkMuted: '#5f5e5e', inkSubtle: '#9a9a9a',
    success:    '#0a9466', warning: '#f5c850', danger: '#e11d2a',
  },
  radius:  { sm: 12, md: 16, lg: 20, xl: 28 },
  spacing: (n: number) => n * 4,
  blur:    { card: 20, bar: 40 },
  // Шрифты — по макетам дизайнера; по умолчанию Inter (UI) + акцидентный для чисел/заголовков.
};
```
Темизация: провайдер темы (light/dark) + хук `useTheme()`. Никаких «магических» цветов в компонентах — только токены.

## 7. Модель данных (Prisma, ключевые сущности)

> Ниже — минимально достаточный набор для MVP. Claude Code детализирует поля/связи в `schema.prisma`.

- **User** — id, phone (уникальный), статус верификации (`MYID_VERIFIED` и т.п.), связь с MyID-данными (ПИНФЛ, ФИО, дата рождения, пол, адрес — read-only), createdAt.
- **Document** — паспорт / водительское удостоверение (kind, серия, номер, дата выдачи, категории), привязка к User (уникальность по `userId+kind`).
- **Vehicle** — авто в «гараже»: марка, модель, год, госномер, VIN/кузов, двигатель, серия/номер техпаспорта, владелец; `userId`; источник данных (НАПП/ручной).
- **Product** — тип (`OSAGO` | `KASKO`), название, описание, активность, страховая компания-эмитент.
- **Tariff / CalcParams** — параметры расчёта (регион, период, коэффициенты, водители, стоимость авто для КАСКО); результат расчёта — премия.
- **Policy** — оформленный полис: номер, тип (OSAGO/KASKO), статус (`DRAFT|ISSUED|ACTIVE|EXPIRED|CANCELLED`), период (start/end), premium, привязка к User + Vehicle, `qrToken` (публичная проверка), файл полиса (ключ в MinIO).
- **Payment** — платёж: сумма, провайдер (payme/click/uzcard), статус (`PENDING|PAID|FAILED|REFUNDED`), привязка к Policy/Wallet, внешний id провайдера.
- **Card** — привязанная карта (токен провайдера, маскированный PAN, тип), `userId`.
- **Wallet** — кошелёк пользователя: баланс; транзакции пополнения/списания.
- **OtpSession** — телефон, код (в dev — фиксированный), срок, попытки.
- **Notification / DeviceToken** — токены устройств (FCM/APNs), журнал уведомлений.

## 8. Мобильное приложение — экраны

### 8.1. Онбординг и авторизация
- **Онбординг** — 2–3 слайда о продукте.
- **Ввод телефона** → **OTP** (dev-код фиксированный, prod — Playmobile).
- **Верификация MyID** — шаг-ап идентификации; после успеха тянем ФИО/ПИНФЛ/адрес → предзаполнение профиля.
- Биометрия/PIN для входа; refresh-токен в secure storage.

### 8.2. Главная (Home)
- Приветствие, быстрые действия (оформить ОСАГО/КАСКО), карточки активных полисов, ближайшее истечение, баланс кошелька.

### 8.3. Каталог → Расчёт → Оформление → Оплата
Пошаговый флоу на `WizardFrame`:
1. **Каталог продуктов** — карточки ОСАГО / КАСКО.
2. **Выбор авто** — из гаража или добавить новое (по техпаспорту → НАПП).
3. **Параметры расчёта** — период, регион, водители (ОСАГО); стоимость/условия (КАСКО).
4. **Результат расчёта** — премия, состав, условия.
5. **Checkout** — проверка данных страхователя/авто, согласия.
6. **Оплата** — карта / кошелёк (Payme/Click/Uzcard).
7. **Успех** — электронный полис с QR, кнопка «Скачать PDF», добавление в «Полисы».

### 8.4. Полисы
- Список (активные/истёкшие), фильтры.
- Деталь полиса: реквизиты, период, статус, **электронный полис + QR**, скачивание PDF.
- Публичная проверка по QR (ссылка `/v/<token>` — открывается без авторизации).

### 8.5. Гараж
- Список авто; добавление по серии/номеру техпаспорта → подтягивание из НАПП; ручное добавление как fallback.
- Деталь авто: реквизиты, связанные полисы; редактирование.

### 8.6. Профиль
- Личные данные (из MyID, read-only): ПИНФЛ, ФИО, дата рождения, пол, адрес.
- Документы: паспорт (заблокирован для верифицированных), водительское удостоверение (редактируемое).
- Язык интерфейса, тема, безопасность (PIN/биометрия), выход.

### 8.7. Финансы
- Кошелёк: баланс, пополнение, история транзакций.
- Карты: список привязанных, добавление/удаление (токенизация у провайдера).
- История платежей по полисам.

### 8.8. Уведомления
- Лента; настройки; deeplink на полис/оплату.

## 9. Бэкенд — модули и эндпоинты (черновой контракт)

> Пути под префиксом `/api`. Все, кроме публичной проверки и авторизации, — под JWT.

| Модуль | Эндпоинты (основные) |
|---|---|
| **auth** | `POST /auth/otp/request`, `POST /auth/otp/verify`, `POST /auth/refresh`, `POST /auth/myid/step-up` |
| **users** | `GET /users/me`, `PATCH /users/me`, `GET /users/me/myid` |
| **documents** | `GET /documents`, `PUT /documents/:kind` |
| **vehicles** | `GET /vehicles`, `POST /vehicles`, `GET/PATCH/DELETE /vehicles/:id`, `POST /vehicles/lookup` (НАПП по техпаспорту) |
| **products** | `GET /products`, `GET /products/:id` |
| **calc** | `POST /calc/osago`, `POST /calc/kasko` → премия |
| **policies** | `POST /policies` (оформить/draft), `POST /policies/:id/issue`, `GET /policies`, `GET /policies/:id`, `GET /policies/:id/pdf`, публичный `GET /public/policies/:token` |
| **payments** | `POST /payments/init`, `POST /payments/:provider/callback` (webhook), `GET /payments/:id` |
| **cards** | `GET /cards`, `POST /cards` (токенизация), `DELETE /cards/:id` |
| **wallet** | `GET /wallet`, `POST /wallet/topup`, `GET /wallet/transactions` |
| **files** | `POST /files/presign-upload`, `GET /files/presign-download` (presigned S3) |
| **notifications** | `POST /notifications/register-device`, `GET /notifications`, воркеры BullMQ |
| **napp** | внутренняя обёртка над НАПП (авто/ВУ/реестр полисов) |

Контракт финализируется по ходу; типы экспортируются в `packages/api-types` из OpenAPI.

## 10. Интеграции (адаптеры + mock-режим)

Каждая интеграция — **интерфейс + две реализации** (`real` и `mock`), выбор через `ENV`. По умолчанию `mock`, чтобы разработка шла без ключей.

| Интеграция | Назначение | ENV-переключатель |
|---|---|---|
| **MyID** | Биометрическая идентификация, ФИО/ПИНФЛ/адрес | `MYID_MODE=mock\|real` + ключи |
| **НАПП** | Данные авто по техпаспорту, ВУ, реестр полисов | `NAPP_MODE=mock\|real` |
| **Платежи** | Payme / Click / Uzcard (провайдер — выбор заказчика) | `PAYMENTS_PROVIDER=…`, `PAYMENTS_MODE=mock\|real` |
| **SMS (OTP)** | Playmobile | `SMS_MODE=mock\|real` (в mock — фиксированный код, напр. `0000`) |

**Требование:** бизнес-логика зависит только от интерфейса, не от конкретного провайдера. Webhooks платежей — идемпотентны и с проверкой подписи.

## 11. Инфраструктура и окружение

- **Локально:** `docker-compose.yml` поднимает Postgres, Redis, MinIO. Prisma-миграции применяются командой.
- **ENV** (`.env`, не коммитить): `DATABASE_URL`, `REDIS_URL`, `MINIO_*`/`S3_*`, `JWT_SECRET`/`JWT_REFRESH_SECRET`, `*_MODE` и ключи интеграций, `PUBLIC_BASE_URL`.
- **Хостинг:** self-hosted (без облачных managed-сервисов) — детали инфраструктуры заказчик определит отдельно; приложение не должно завязываться на конкретного облачного провайдера.

## 12. Безопасность и нефункциональные требования

- JWT access (короткий) + refresh (ротация); refresh — в secure storage на устройстве.
- Секреты — только через ENV; в репозиторий не попадают.
- Загрузка файлов — через **presigned POST** напрямую в S3 (креды хранилища клиенту не отдаются).
- На устройстве — encrypted MMKV для данных, keychain для токенов.
- Валидация всего ввода (class-validator/Zod); rate-limit на OTP.
- Логи — структурные JSON; без утечки ПДн в логи.
- Производительность: списки — пагинация; изображения — сжатие; офлайн-устойчивость базовых экранов.
- Доступность: контраст, размеры тач-таргетов, поддержка динамического шрифта.

## 13. Фазы реализации (роадмап для Claude Code)

Каждая фаза — самостоятельный, проверяемый инкремент.

### Фаза 0 — Каркас
**Deliverables:** монорепо (pnpm+turbo), `apps/mobile` (Expo) + `apps/api` (NestJS) собираются и запускаются; `docker-compose` (Postgres/Redis/MinIO); Prisma подключён; корневой + локальные `CLAUDE.md`; `docs/QUESTIONS.md`.
**Приёмка:** `pnpm dev` поднимает мобильное и API; health-эндпоинт отвечает; пустое приложение открывается на устройстве/эмуляторе.

### Фаза 1 — Дизайн-система + i18n + навигация
**Deliverables:** компоненты Liquid Glass (раздел 5), темы light/dark, `packages/i18n` (4 локали, каркас ключей), корневая навигация (табы: Home/Полисы/Гараж/Профиль + модальные флоу).
**Приёмка:** демо-экран показывает все базовые компоненты в обеих темах; переключение языка работает.

### Фаза 2 — Авторизация + профиль
**Deliverables:** OTP-флоу (SMS mock), MyID step-up (mock) с предзаполнением, экран Профиль (личные данные + документы), JWT+refresh, безопасное хранение.
**Приёмка:** пользователь входит по OTP, «проходит» MyID (mock), видит профиль; повторный вход по биометрии/PIN.

### Фаза 3 — Гараж + НАПП
**Deliverables:** список/деталь/добавление авто; `POST /vehicles/lookup` (НАПП mock) с подтягиванием данных по техпаспорту; ручной fallback.
**Приёмка:** авто добавляется по техпаспорту (mock-данные) и вручную; отображается в гараже.

### Фаза 4 — Оформление ОСАГО (сквозной сценарий)
**Deliverables:** каталог → выбор авто → расчёт ОСАГО → checkout → оплата (mock) → выпуск полиса с QR + PDF; раздел «Полисы»; публичная проверка `/public/policies/:token`.
**Приёмка:** полный путь до выпущенного е-полиса ОСАГО с работающим QR и скачиванием PDF.

### Фаза 5 — КАСКО
**Deliverables:** расчёт и оформление КАСКО (параметры стоимости/условий), переиспользуя флоу Фазы 4.
**Приёмка:** сквозной сценарий КАСКО.

### Фаза 6 — Финансы
**Deliverables:** кошелёк (баланс/пополнение/транзакции), карты (токенизация mock), история платежей.
**Приёмка:** пополнение кошелька, оплата полиса из кошелька/картой, корректная история.

### Фаза 7 — Уведомления + полировка
**Deliverables:** push (FCM/APNs) — напоминания об истечении/статусы оплаты; пустые/ошибочные состояния; тесты ключевых сценариев; подготовка к сборкам.
**Приёмка:** приходит push об истечении полиса; критичные сценарии покрыты тестами.

### Фаза 8 — Реальные интеграции (по мере получения ключей)
Переключение `*_MODE=real` для MyID, НАПП, платежей, SMS; проверка webhooks и подписей.

## 16. Критерии приёмки MVP (Definition of Done)

- [ ] Пользователь регистрируется по OTP и проходит MyID.
- [ ] Добавляет авто в гараж (по техпаспорту через НАПП или вручную).
- [ ] Оформляет **ОСАГО** и **КАСКО** от каталога до оплаты.
- [ ] Получает **электронный полис с QR** и скачивает **PDF**; QR проверяется публично.
- [ ] Оплачивает картой и из кошелька; видит историю платежей.
- [ ] Профиль показывает данные MyID и документы.
- [ ] Приложение работает на **4 локалях**, в **светлой и тёмной** темах, в стиле **Liquid Glass**.
- [ ] Интеграции переключаются `mock ↔ real` через ENV.
- [ ] Секреты не в репозитории; ключевые сценарии покрыты тестами.
