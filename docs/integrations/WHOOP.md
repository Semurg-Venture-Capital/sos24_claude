# WHOOP.md — Анализ интеграции с WHOOP (носимый трекер здоровья)

> **WHOOP** — американский носимый фитнес-браслет (без экрана) с подписочной моделью.
> Отслеживает восстановление, сон, нагрузку, ВСР, пульс, дыхание, температуру кожи, SpO₂.
> Для платформы SOS24 — источник **актуальных биометрических данных** в модуль «Здоровье» (M14).
>
> Developer Platform: `https://developer.whoop.com/`
> Портал разработчика (регистрация приложения): `https://developer-dashboard.whoop.com/`
> REST base URL: `https://api.prod.whoop.com`
> Актуальная версия API: **v2** (v1 выведен из эксплуатации, вебхуки v1 удалены).
>
> **Последнее обновление:** 2026-07-08
> **Статус:** исследование, интеграция НЕ начата. Ждём решения руководства (см. §9).

---

## 1. Что даёт эта интеграция SOS24

WHOOP — источник **непрерывно обновляемых** показателей здоровья пользователя, которые
приложение SOS24 само собирать не умеет (нужен носимый датчик). Что это даёт продукту:

| Сценарий в SOS24 | Данные WHOOP |
|---|---|
| Экран «Мои показатели» в разделе «Здоровье» — актуальная сводка организма | Recovery %, ВСР, пульс покоя, качество сна, нагрузка, дыхание |
| Обогащение **ИИ-триажа** контекстом («сегодня низкое восстановление, ВСР ниже базовой») | Recovery + ВСР + пульс покоя + сон |
| Авто-заполнение мед.карты (рост/вес) | Body Measurement |
| Базовые витальные показатели при **SOS/ЧП** (для диспетчера/врача) | пульс покоя, ВСР, дыхание, SpO₂ |
| Долгосрочный тренд здоровья (профилактика, ДМС-скоринг в перспективе) | вся история |

> ⚠️ **Ключевая оговорка бизнес-модели.** WHOOP требует **физического браслета + активной
> платной подписки** у пользователя. В Узбекистане это очень нишевая аудитория. Реальная
> ценность интеграции напрямую зависит от того, сколько наших целевых пользователей носят WHOOP.
> Это первый вопрос к руководству (см. §9, В-1).

## 2. Модель данных — что именно можем вытащить

WHOOP **не отдаёт «сырой» живой поток** (нет посекундного пульса через публичный API).
Данные приходят **готовыми дискретными объектами** после обработки в облаке WHOOP.

### 2.1. Recovery (готовность организма) — обновляется 1×/сутки утром
| Поле | Смысл |
|---|---|
| `recovery_score` | Восстановление 0–100 % (главная метрика WHOOP) |
| `resting_heart_rate` | Пульс покоя (уд/мин) |
| `hrv_rmssd_milli` | Вариабельность сердечного ритма (ВСР, мс) |
| `spo2_percentage` | Сатурация кислорода¹ |
| `skin_temp_celsius` | Температура кожи¹ |
| `user_calibrating` | Идёт ли ещё калибровка новичка |
| `score_state` | `SCORED` / `PENDING_SCORE` / `UNSCORABLE` |

### 2.2. Sleep (сон) — после каждого сна/дневного сна
| Поле | Смысл |
|---|---|
| `total_in_bed_time_milli` | Время в кровати |
| `total_light_sleep_time_milli` | Лёгкий сон |
| `total_slow_wave_sleep_time_milli` | Глубокий сон |
| `total_rem_sleep_time_milli` | REM-фаза |
| `total_awake_time_milli` | Бодрствование |
| `sleep_performance_percentage` | Эффективность сна vs потребность |
| `sleep_efficiency_percentage` | Доля сна во времени в кровати |
| `sleep_consistency_percentage` | Регулярность режима |
| `respiratory_rate` | Частота дыхания (вдохов/мин) |
| `sleep_cycle_count`, `disturbance_count` | Циклы сна, пробуждения |
| «Потребность в сне» | база + долг сна + от нагрузки + от напов (в мс) |

### 2.3. Cycle (физиологические сутки / нагрузка) — обновляется в течение дня
| Поле | Смысл |
|---|---|
| `strain` | Дневная нагрузка 0–21 (шкала WHOOP) |
| `kilojoule` | Энергозатраты |
| `average_heart_rate`, `max_heart_rate` | Средний/макс. пульс за сутки |

### 2.4. Workout (тренировка) — после каждой тренировки
`sport_name`, `strain`, `average_heart_rate`, `max_heart_rate`, `kilojoule`,
`percent_recorded`, `distance_meter`, `altitude_gain_meter`, `altitude_change_meter`,
`zone_durations` (5 пульсовых зон в мс).

### 2.5. Body Measurement — редко
`height_meter`, `weight_kilogram`, `max_heart_rate`.

### 2.6. Profile
`user_id`, `email`, `first_name`, `last_name`.

¹ **SpO₂ и температура кожи — только для браслетов WHOOP 4.0 и новее.**

## 3. Насколько быстро можем обновлять и как

### 3.1. Реальная свежесть данных
Цепочка появления данных:
1. Браслет по Bluetooth синхронизируется с приложением WHOOP на телефоне (в фоне / при открытии).
2. Облако WHOOP обрабатывает и «оценивает» (`score_state → SCORED`).
3. → срабатывает **вебхук** нам → мы забираем объект.

**Достижимая свежесть:** «как только браслет пользователя синхронизировался и WHOOP посчитал» —
обычно **минуты** после события. Не секунда-в-секунду. Recovery — раз в сутки утром. Сон и
тренировки — после завершения активности и синхронизации браслета.

### 3.2. Механизм обновления (используем оба)
- **Вебхуки (push, основной путь).** Подписка на 6 событий:
  `recovery.updated` / `recovery.deleted`, `sleep.updated` / `sleep.deleted`,
  `workout.updated` / `workout.deleted`.
  Payload: `{ user_id, id, type, trace_id }` (POST по HTTPS). По `id` дёргаем объект и сохраняем.
  Подпись: `base64(HMAC-SHA256(timestamp_header + raw_body, client_secret))`, сверяем с заголовком
  `X-WHOOP-Signature` (+ `X-WHOOP-Signature-Timestamp`). При не-2XX — WHOOP ретраит 5 раз за ~час.
- **Периодический поллинг (fallback / бэкфилл).** Крон-джоб раз в N часов подтягивает историю (при
  первом подключении и на случай пропущенных вебхуков). Переиспользуем существующую очередь **BullMQ**.

### 3.3. Лимиты API
- **100 запросов/мин** и **10 000 запросов/сутки** — **на всё приложение** (не на пользователя).
- Превышение → `429 Too Many Requests`. Можно запросить повышение через форму.
- Вывод: при вебхук-архитектуре расход минимальный; поллинг надо бюджетировать под число юзеров.

### 3.4. Стоимость и лимит пользователей (production)
- **API WHOOP полностью бесплатный** — и в разработке, и в проде. «Платного доступа» как такового нет.
- **До одобрения приложения:** максимум **10 членов WHOOP**.
- **После одобрения:** доступ ко **всем членам WHOOP — лимита пользователей нет** (тоже бесплатно).
- **Обязательное условие:** у разработчика должен быть **свой браслет WHOOP + активная подписка**.
- Итог: единственная плата — это сам браслет/подписка на стороне **конечного пользователя**, не у нас.

## 4. Аутентификация (OAuth 2.0)

- **Flow:** Authorization Code + скоуп `offline` (иначе не выдаётся refresh-token).
- **Скоупы:** `read:recovery`, `read:sleep`, `read:workout`, `read:cycles`, `read:profile`,
  `read:body_measurement`, `offline`.
- **Access-token:** живёт ~1 час (`expires_in = 3600`).
- **Refresh-token:** ротируется при каждом обновлении (старый инвалидируется) — храним **последний**.
- **URLs:** авторизация `…/oauth/oauth2/auth`, обмен кода/refresh `…/oauth/oauth2/token`.
- **В мобилке:** OAuth через in-app browser (`expo-auth-session`/`expo-web-browser`) + возврат по deep-link.
- **Хранение токенов:** зашифрованно на пользователя — переиспользуем `MED_ENCRYPTION_KEY` + field-cipher
  (тот же механизм, что для мед.карты).

## 5. ⚠️ Ограничения и риски

| # | Ограничение | Влияние |
|---|---|---|
| 1 | **Лимит 10 пользователей** до одобрения приложения WHOOP | Пилот/демо — сразу; публичный запуск — только после ревью |
| 2 | **Ревью приложения раз в месяц** | Заявку надо подавать заранее; нужны App Name, Privacy Policy URL, соответствие бренд-гайдам WHOOP |
| 3 | **Нужен браслет WHOOP + платная подписка** у юзера | Нишевая аудитория в Узбекистане — бизнес-риск ценности |
| 4 | Нет live-HR (посекундного пульса) | «Актуально» = после синхронизации браслета, не realtime |
| 5 | SpO₂ и темп. кожи — только WHOOP 4.0+ | Часть полей будет пустой у старых браслетов |
| 6 | Данные — за пределами РУз (облако WHOOP, США) | Юр. вопрос по обработке мед-данных граждан РУз (см. §9, В-6) |

## 6. План интеграции на нашей стороне (черновик)

**Модель данных (Prisma):**
- `WearableConnection` — `provider` (WHOOP), `userId`, зашифрованные `accessToken`/`refreshToken`,
  `expiresAt`, `scopes`, `whoopUserId`, `status`.
- Кэш-таблицы `WhoopRecovery` / `WhoopSleep` / `WhoopCycle` / `WhoopWorkout` (или обобщённая
  `HealthMetricDaily`) — храним последние + историю.

**Бэкенд** — новый подмодуль `apps/api/src/health/wearables/whoop/`:
- `GET /health/wearables/whoop/connect` → OAuth URL; `…/callback` → сохранить токены.
- Приёмник вебхуков `POST /webhooks/whoop` (проверка подписи → джоба в BullMQ → фетч+сохранение).
- Крон-бэкфилл + авто-refresh токенов.

**Мобилка** — раздел «Здоровье»:
- Кнопка «Подключить WHOOP» (OAuth-флоу).
- Карточка «Мои показатели»: Recovery %, ВСР, пульс покоя, качество сна, Strain, дыхание + разбивка сна.
- Авто-обновление по вебхукам (наш push уже есть — можно уведомлять «данные обновлены»).

**Синергия с существующим:** авто-заполнение роста/веса в мед.карту; контекст для ИИ-триажа;
витальные показатели для SOS.

**Оценка объёма (черновая):** бэкенд ≈ 2–3 дня, мобилка ≈ 2 дня, заявка на approval — параллельно.

## 7. Альтернатива — агрегатор носимых устройств

Вместо прямой интеграции только с WHOOP можно подключить **агрегатор** (Junction / Terra / Rook) —
один API нормализует WHOOP + Apple Watch + Garmin + Samsung + Fitbit и т.д.

| | Прямая WHOOP | Агрегатор |
|---|---|---|
| Охват устройств | только WHOOP | десятки вендоров разом |
| Скорость первого запуска | быстрее (1 интеграция) | чуть дольше настройка, зато сразу все |
| Стоимость | бесплатный API WHOOP | платный SaaS (против self-hosted-приоритета проекта) |
| Зависимость | от WHOOP | от посредника |
| Лимит 10 юзеров WHOOP | остаётся | может частично сниматься (через их партнёрство) |

> У нас уже есть импорт из **Apple HealthKit** (группа крови/рост/вес). WHOOP его дополнит.
> Если цель — «любой браслет», агрегатор стратегически выгоднее. Решение — за руководством (В-2).

### Конкретные агрегаторы (WHOOP + Google + Samsung Watch + Apple + Garmin через один API)

| Платформа | Охват | Модель / цена | Особенность |
|---|---|---|---|
| **Terra** | 500+ вендоров | Free tier (вкл. WHOOP + Samsung), далее от **$399/мес** годовой / $499 помесячно, 100k кредитов; ~200 кредитов/юзер → **~$4 800/год за 500**, ~$10 800/год за 1000 | Фокус на фитнес/активность |
| **ROOK** | 400+ | Per-user SaaS | «Клинического качества», нормализация |
| **Junction** (экс-Vital) | Основные вендоры | Enterprise SaaS (по договору) | Dev-first + лаб-анализы |
| **Spike** | 300+ | Per-user SaaS | Реалтайм-доступ |
| **Open Wearables** | «любое устройство» | **Self-hosted, MIT, без платы за юзера** | Совпадает с self-hosted-приоритетом SOS24 |

**Ориентир рынка:** SaaS-агрегаторы — **$0.50–$2.00 за пользователя/мес**; кастомная интеграция с нуля — $65k–140k.

**Нюансы Google/Samsung:**
- **Google Fit устарел** → актуально **Google Health Connect** (хаб данных на Android).
- Данные **Samsung Watch** идут через **Samsung Health** → Health Connect.
- Все крупные агрегаторы это поддерживают: **один агрегатор = WHOOP + Google + Samsung + Apple + Garmin разом**.

## 8. Итоговая оценка (кратко)

- **Технически** — интеграция несложная и хорошо документированная (OAuth + вебхуки + BullMQ, всё
  переиспользуется из уже готового). Реализуемо за ~1 неделю на пилот.
- **Свежесть** — «почти реального времени» на уровне минут после синхронизации браслета; постоянного
  живого пульса нет (ограничение платформы WHOOP, не наше).
- **Главный вопрос — не технический, а бизнесовый:** сколько целевых пользователей реально носят
  WHOOP, и оправдывает ли это интеграцию против альтернативы «агрегатор для всех браслетов».

## 9. Открытые вопросы (к руководству / заказчику)

> Детальная управленческая версия этих вопросов — в `docs/integrations/WHOOP_BRIEF.md` (для пересылки).

| # | Вопрос | Почему важно |
|---|---|---|
| В-1 | Какова доля наших целевых пользователей, реально носящих WHOOP? | Определяет саму целесообразность интеграции |
| В-2 | Прямая интеграция **только WHOOP** или **агрегатор** (WHOOP + Apple Watch + Garmin + …)? | Разная стоимость, охват и сроки |
| В-3 | Масштаб на старте: **пилот ≤10 браслетов** (сразу) или **публичный запуск** (нужна заявка на approval заранее)? | Влияет на сроки и порядок действий |
| В-4 | Какие данные приоритетны: «здоровье» (Recovery/ВСР/сон/пульс) или ещё и «фитнес» (Strain/тренировки)? | Определяет объём первого релиза |
| В-5 | Есть ли у нас/заказчика **аккаунт разработчика WHOOP** и хотя бы **один браслет** для теста? | Без этого нельзя даже начать пилот |
| В-6 | Юридически: допустимо ли хранение биометрии граждан РУз в облаке WHOOP (США)? Нужно ли согласие/локализация? | Требования к обработке мед-данных |
| В-7 | Монетизация: интеграция — бесплатная фича, часть ДМС-пакета или платная опция? | Влияет на приоритет и модель |
