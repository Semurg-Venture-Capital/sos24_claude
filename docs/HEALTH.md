## 0. Резюме и решения

Новый раздел приложения — **5-я вкладка нижнего меню «Здоровье»** (`Главная · Полисы · Здоровье · Гараж · Профиль`). В коде сейчас **ничего нет**: ни `features/health`, ни бэкенд-модуля, ни LLM. Дизайн — production-качества, в стиле liquid-glass + красный акцент (совпадает с текущим приложением).

**Зафиксированные решения (обсуждено 2026-07-02):**
- **ИИ-триаж** — на старте **mock-адаптер** (сценарные ответы). Реальный LLM подключаем позже через тот же интерфейс. Движок LLM (Claude/GPT) — выбор отложен.
- **Видео-консультации** — **отложены**. Экраны M14.7/M14.8 и опция «видео-приём» делаем как UI-заглушку «Скоро»; запись к врачу — только **очно**.
- **Объём первой итерации** — весь раздел M14 (все 12 экранов UI), но с учётом двух пунктов выше.
- **Медицинские данные** — храним **на нашем бэкенде с шифрованием чувствительных полей** + явное согласие пользователя.

**Оценка объёма:** отдельный крупный модуль, сопоставимый с «европротокол + партнёры». Самые тяжёлые/рисковые куски — ИИ-триаж (когда дойдёт до реального LLM) и видео (отложены).

## 4. Модель данных (Prisma, новые сущности)

### 4.1. Мед.карта (Medical ID) — чувствительные данные
```
model MedicalProfile {
  id           String   @id @default(cuid())
  userId       String   @unique
  user         User     @relation(...)
  bloodType    String?           // группа крови + резус (напр. "O(I) Rh+")
  heightCm     Int?
  weightKg     Int?
  allergies    String?  @db.Text // ШИФРУЕТСЯ (JSON-массив в открытом виде до шифрования)
  chronic      String?  @db.Text // ШИФРУЕТСЯ — хронические заболевания
  medications  String?  @db.Text // ШИФРУЕТСЯ — постоянные лекарства
  organDonor   Boolean?          // донор органов
  pregnancy    Boolean?
  dmsPolicy    String?           // полис ДМС/ОМС
  doctorName   String?           // лечащий врач
  consentAt    DateTime?         // согласие на обработку мед-данных
  updatedAt    DateTime @updatedAt
  @@map("medical_profiles")
}
```
**Шифрование:** поля `allergies/chronic/medications` (и при необходимости другие «диагнозные») — **field-level шифрование** (AES-256-GCM, ключ из `MED_ENCRYPTION_KEY` в ENV). В проекте crypto-утилиты пока НЕТ — завести `common/crypto/field-cipher.ts` (encrypt/decrypt) и использовать в сервисе (шифруем на запись, дешифруем на чтение владельцу). В БД — только шифротекст.

### 4.2. Экстренные контакты
```
model EmergencyContact {
  id        String   @id @default(cuid())
  userId    String
  user      User     @relation(...)
  name      String
  relation  String?           // «брат», «мама» …
  phone     String
  sortOrder Int      @default(0)
  createdAt DateTime @default(now())
  @@index([userId])
  @@map("emergency_contacts")
}
```
Лимит — 3 контакта на пользователя (валидация в сервисе).

### 4.3. Врач (при клинике-партнёре)
```
model Doctor {
  id           String   @id @default(cuid())
  partnerId    String?          // клиника (Partner), может быть null для частного
  partner      Partner? @relation(...)
  fullName     String
  specialty    String           // Терапевт / ЛОР / Кардиолог …
  experienceY  Int?
  bio          String?  @db.Text
  photoKey     String?          // MinIO
  rating       Float    @default(0)
  reviewCount  Int      @default(0)
  pricePrimary Int?             // первичный приём, сум
  priceRepeat  Int?             // повторный
  priceVideo   Int?             // видео (для будущего)
  videoEnabled Boolean  @default(false) // видео-приём (пока false везде)
  verified     Boolean  @default(false) // «Партнёр SOS24» / badge-check
  active       Boolean  @default(true)
  createdAt    DateTime @default(now())
  @@index([partnerId]) @@index([specialty])
  @@map("doctors")
}
```
Запись к врачу — переиспользуем `PartnerBooking` (добавить опциональный `doctorId`), либо отдельная `DoctorAppointment` (решить на этапе C; предпочтительно расширить `PartnerBooking`, чтобы не плодить логику записей). Слоты — по `durationMin`, как в партнёрах.

### 4.4. ИИ-триаж (сессия)
```
model TriageSession {
  id         String   @id @default(cuid())
  userId     String
  user       User     @relation(...)
  messages   Json     // история [{role, text, at}]
  symptoms   Json?    // распознанные симптомы (чипы)
  verdict    String?  // предв. диагноз (текст)
  urgency    String?  // low | medium | high
  confidence Int?     // 0..100
  createdAt  DateTime @default(now())
  @@index([userId, createdAt])
  @@map("triage_sessions")
}
```
Историю триажа тоже относим к чувствительным данным (шифровать `messages/symptoms/verdict` — по тем же правилам, либо на уровне доступа «только владелец»).

## 5. Бэкенд — модуль `health`

Новый модуль `apps/api/src/health/`. Все эндпоинты под JWT (кроме внутренних вызовов провайдеров). DTO + валидация на каждом.

| Область | Эндпоинты |
|---|---|
| **Мед.карта** | `GET /health/medical-profile`, `PUT /health/medical-profile` (+ фиксация `consentAt` при первом сохранении) |
| **Экстренные контакты** | `GET /health/emergency-contacts`, `POST` (лимит 3), `PATCH/DELETE /:id` |
| **Врачи/клиники** | `GET /health/doctors` (поиск/фильтр по специальности/клинике), `GET /health/doctors/:id`, `GET /health/clinics` (= partners категории «медицина») |
| **Запись** | `POST /health/appointments` (очно), `GET /health/appointments`, `GET /health/doctors/:id/slots` — поверх `PartnerBooking` |
| **ИИ-триаж** | `POST /health/triage/start`, `POST /health/triage/:id/message` (ответ mock-провайдера), `POST /health/triage/:id/finalize` (вердикт+срочность) |
| **ЧП / SOS** | `POST /health/sos/trigger` (создать тревогу, оповестить контакты, принять геолокацию), `POST /health/sos/:id/cancel` |

### 5.1. Адаптер триажа (ключевое)
Интерфейс `TriageProvider` c методами `ask(session, userMessage)` и `finalize(session)`. Реализации:
- `MockTriageProvider` (сейчас) — сценарий вопрос-ответ (температура? боль в горле? …) + правила для вердикта/срочности/уверенности. Захардкоженные быстрые ответы и «диагнозы» из дизайна (напр. «Острый ринит», срочность низкая, 78%).
- `LlmTriageProvider` (позже) — реальный LLM (Claude/GPT) с медицинским промптом; тот же интерфейс. Переключение через `TRIAGE_MODE=mock|llm` в ENV.
**Обязательно:** дисклеймер «это не диагноз» в ответах и на экране; красный «вызовите 103» при высокой срочности.

## 6. Экраны — детали из дизайна (что рисуем)

### M14.1 Хаб
SOS-герой (градиент, пульс-анимация «удерживайте»), тёмная карточка «ИИ-диагноз» → триаж, сетка 2×2 быстрых плиток (Мед.карта / Скорая 103 / Найти врача / Контакты ЧП), секция «Врачи рядом» (2 карточки `MedDoctorCard`), таб-бар (Health активна).

### M14.2 ИИ-триаж
Хедер «SOS24 · Медицинский ИИ · анализирует симптомы», лента сообщений (AI-карточки, голосовое сообщение пользователя с волной, быстрые ответы-чипы, индикатор «печатает»), композер с камерой + полем + **крупной кнопкой микрофона**. Плашка «Это не диагноз».

### M14.3 Предв. диагноз
Тёмная карточка вердикта: «Предварительно» + бейдж срочности + название + описание + **шкала уверенности %**. Секции «Вы сообщили» (чипы симптомов), «Что рекомендуем» (строки, красная строка про 103), дисклеймер. Sticky CTA: «Записаться к врачу» + «Видео сейчас» (**заглушка**).

### M14.4 Врачи и клиники
Поиск, фильтр-чипы (Все/Терапевт/ЛОР/Кардиолог/Фильтры), метка «Партнёрские клиники · Ташкент», карточки врачей (`MedDoctorCard`: аватар+verified, спец+стаж, рейтинг+отзывы, дистанция, «видео-приём» бейдж, цена, «Записаться»), карточка мед.центра.

### M14.5 Профиль врача
Шапка (аватар, verified, ФИО, спец, теги «Партнёр SOS24»/«Видео-приём»), статы (рейтинг/опыт/приёмов), «О враче», «Услуги и цены», «Ближайшая запись» (сетка слотов), «Отзывы». Sticky: «Записаться очно» + «Видео» (**заглушка**).

### M14.6 Бронирование
Мини-карточка врача, формат приёма (Segmented — очно/видео; **видео задизейблено**), выбор даты/времени, → оплата (переисп用ать `payments`).

### M14.7/14.8 Видео — **заглушки «Скоро»**
Экраны реализуются визуально, но с оверлеем/состоянием «Видео-консультации скоро». Реальный Meet/WebRTC — вторая итерация.

### M14.9 Мед.карта
Шапка (имя, пол, возраст, дата рождения), виталы (кровь/рост/вес), секции «Аллергии» (чипы), «Хронические», «Постоянные лекарства», «Донор/беременность», «Полис ДМС», «Лечащий врач». Табы «Данные / Экстренные контакты / Медкарта». Кнопка редактирования.

### M14.10 Редактирование
Форма: ФИО, дата рождения, группа крови, рост/вес, аллергии, хронические, лекарства, полис ОМС/ДМС, лечащий врач. «Сохранить». Согласие на обработку мед-данных при первом сохранении.

### M14.11 Экстренные контакты
Список (до 3), карточка контакта (аватар, имя, отношение, телефон, кнопка звонка), «Добавить контакт», блок «При срабатывании SOS».

### M14.12 Экран ЧП / SOS
Активная тревога: «Отправляем помощь…», «Оповещаем близких и передаём геолокацию диспетчеру», отмена. Интеграция с геолокацией + оповещением (push/SMS контактам).

## 8. Безопасность и приватность (критично для мед-данных)

- **Field-level шифрование** чувствительных полей (`allergies/chronic/medications`, история триажа) — AES-256-GCM, ключ в ENV (`MED_ENCRYPTION_KEY`), не в репозитории. Завести `common/crypto/field-cipher.ts`.
- **Согласие пользователя** на обработку мед-данных (фиксировать `consentAt`) — до первого сохранения мед.карты.
- Доступ к мед-данным — **только владелец** (guard по `userId`); в логи ПДн не писать.
- **Дисклеймеры ИИ** обязательны на экранах триажа/диагноза («не диагноз, точный ставит врач»).
- SOS/геолокация — передавать только при явном срабатывании тревоги.
