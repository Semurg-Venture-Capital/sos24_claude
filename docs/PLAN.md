## PIVOT 2026-06-02 — Расширение продукта

> **Статус:** зафиксировано, влияет на всю архитектуру и маркетинг.

### Что изменилось
- **Было:** платформа для авто-страхования (ОСАГО/КАСКО)
- **Стало:** универсальная платформа «SOS в любой ситуации» — страхование всех видов + ИИ-ассистент + здравоохранение + умные часы

### Новые продукты и фичи

#### 1. Все виды страхования (расширение каталога)
| Вид | Статус |
|---|---|
| ОСАГО | ✅ готово |
| КАСКО | ✅ готово |
| Медицинское страхование | 🔲 нужно добавить |
| Страхование жизни | 🔲 нужно добавить |
| Имущество (квартира, дом) | 🔲 нужно добавить |
| Путешествия | 🔲 нужно добавить |

#### 2. ИИ-чатбот (LLM-маршрутизатор)
- **Реализация:** Claude API / OpenAI API — интегрируем сами в бэкенд
- **Функции:**
  - Принять запрос пользователя на естественном языке
  - Определить категорию: SOS / страхование / медицина / партнёр
  - Маршрутизировать в нужный раздел приложения или специалиста
  - Первичный анализ симптомов + направление к врачу
  - Бронирование процедур и консультаций (платных/бесплатных)
  - Инициация SOS-звонка

#### 3. Мгновенная выплата 2 млн сум при ДТП
- **Механика:** SOS24 платит из собственного резерва в момент ДТП (подтверждение через аджастера), затем получает возмещение от страховой компании
- **Условия:** активный полис + подтверждение факта ДТП аджастером + лимит 2 млн сум
- **Финансовая модель:** SOS24 = финансовый буфер между клиентом и страховой
- **Статус финансовой схемы:** нужно юридическое оформление (открытый вопрос)

#### 4. Здравоохранение
- **Без очереди:** партнёрские клиники принимают клиентов SOS24 вне очереди (травматология и другие)
- **Цифровая медицинская книжка:** интеграция с Apple Health / Android Health Connect — хранение данных о прививках, диагнозах, лекарствах
- **Статус интеграции с госсистемой здравоохранения РУз:** открытый вопрос (Q-new)

#### 5. Умные часы (SOS-приложение)
- **Платформы:** Apple Watch (watchOS) + Samsung Galaxy Watch (Wear OS)
- **Функция:** одна кнопка SOS с запястья → вызов помощи + геолокация
- **Фаза:** после основного мобильного приложения
- **Статус:** концепция, детали не определены

## 0. Зафиксированные решения (history of record)

> Решения, по которым была развилка и ответ получен. Хранятся здесь, чтобы не потерять. Источник правды для этого раздела — `PLAN.md`; `QUESTIONS.md` содержит только **открытые** вопросы.

### Бренд и юридическое
- **Бренд:** SOS24, **домен:** sos24.uz *(2026-05-05)*
- **Юр. лицо страховой и лицензии:** будут переданы клиентом позже — не блокирует разработку, блокирует только публикацию полисов в продакшене.
- **Логотип:** делается дизайнером (нейминг утверждён, визуал — за дизайнером).

### Госинтеграции
- **NAPP** = Национальное агентство перспективных проектов РУз (napp.uz) — регулятор страхового рынка.
- Через NAPP работают **все три** госинтеграции: база авто, реестр водительских удостоверений, реестр электронных полисов.
- Контакт со стороны заказчика: **Хожиакбар** (он на стороне клиента, координирует интеграцию).

### Идентификация
- **Старт:** SMS-OTP (Playmobile) + MyID + OneID — реализуем все три.
- **e-IMZO:** фаза 2, по остаточному принципу (не блокирует MVP) *(2026-05-05)*.
- SMS-провайдер — **Playmobile**, контакт **Жамшид** (на стороне клиента).

### Платежи
- **Свой платёжный шлюз** — клиент строит **собственный gateway**: ввод номера карты в нашем UI, списание через Uzcard rails *(2026-05-05)*.
- **Click / Payme / Uzum: НЕ используем** — отрезаны, шлюз свой.
- Возврат при аннулировании — **100% полный**.
- Рассрочка — **своя** (страховая) + внешние BNPL (Uzum Nasiya / Alif / Anor — по списку, уточняется).
- ⚠️ **PCI-DSS:** ввод номера карты в собственном UI = scope PCI-DSS. Нужна либо сертификация уровня SAQ A-EP/D, либо токенизация на стороне Uzcard (данные карты идут напрямую к Uzcard, у нас — только токен). Решение по технической реализации — отдельный архитектурный документ.

### Тарифы
- ОСАГО — гос. база + надбавки страховой (двухуровневый конструктор).
- КАСКО — формула передаст андеррайтер **Махмуд-ака (заказчик)**.
- Управление тарифами — менеджер через UI админки, переменные коэффициенты.

### Урегулирование убытков
- Европротокол — электронный, через мобильное приложение SOS24.
- Вызов аварийного комиссара — **да, нужен**.
- SLA по умолчанию — **60** (предположительно дней), переменно по тарифу.

### Партнёрская сеть (B2B)
- **Делаем отдельный кабинет** для партнёров (СТО, медклиники) *(2026-05-05)*.
- Партнёры выкладывают свои услуги/активы, получают клиентов от страховой.
- ⚠️ **Медклиники** в авто-страховой — нестандартное расширение; точный сценарий использования (страхование пассажиров / медосмотр для ВУ / медпомощь после ДТП) пока **не определён** — решаем по ходу.

### Локализация
- **4 локали:** uz-Latn, uz-Cyrl, ru, en (узбекский латиница и кириллица — две независимые локали).

### Юридическое
- Юрист — у заказчика. Он пишет оферту, политику, условия.

### Аналитика и инфраструктура
- **PostHog** — аналитика поведения.
- **Sentry** — crash-reporting.
- **Self-hosted** инфраструктура: всё разворачиваем на серверах заказчика. Без облачных SaaS.

### Запуск и операции
- Публикацию в App Store / Google Play — делает **клиент** (его аккаунты разработчика).
- Поддержка — **свой call-центр 24/7** на стороне клиента.
- Бухгалтерия — интеграция с **1С**.

### Дизайн-процесс
- Референсы дизайнеру переданы. Дизайнер работает самостоятельно, результат — дизайн-система и макеты в Figma.

## 1. Цель

Запустить в Узбекистане платформу **SOS24** — автостраховой сервис «помощник 24/7 на дороге»: продажа полисов (ОСАГО, КАСКО), электронный полис с QR, онлайн-урегулирование убытков, вызов аварийного комиссара, партнёрская сеть (СТО, медклиники).

Платформа состоит из:

1. **Мобильное приложение** (React Native, iOS + Android) — для клиентов
2. **Веб-лендинг** sos24.uz — для маркетинга и привлечения трафика
3. **Веб-админка** — для сотрудников страховой
4. **Веб-кабинет партнёра** — для СТО, медклиник, других B2B-партнёров (новый модуль)
5. **Бэкенд** — единое API для всех клиентов

**Принципы:**
- **Self-hosted** инфраструктура (без Vercel / Heroku / managed-облаков): разворачиваемся на серверах заказчика. Подтверждено, см. `QUESTIONS.md` Q8.3.
- **Монорепо** через Turborepo + pnpm: общие TS-типы между mobile/web/api/partner.
- Бэкенд: **Node.js (NestJS) + PostgreSQL + Redis + S3-совместимое (MinIO)** — всё ставится локально на серверах заказчика.
- API: REST + OpenAPI; WebSocket — для статусов выплат / чата поддержки / диспетчеризации комиссара.
- Очереди: **BullMQ (Redis)** — SMS, push, генерация PDF, обработка фото, синк с 1С, синк с NAPP.

## 6. Принципы разработки

- **Type-safe end-to-end:** общие TS-типы между mobile/web/api (через монорепо)
- **API-first:** OpenAPI-спека пишется до реализации
- **i18n с первого дня** — никаких хардкод-строк, **4 локали**
- **Платёжная безопасность:** PCI-чувствительные данные не хранятся у нас, токенизация на стороне Uzcard / прочих эквайеров
- **PII под защитой:** паспорт/водительское — шифруются at-rest
- **Audit-log на всё критичное:** покупка полиса, выплата, изменение тарифа, действия партнёров
- **Feature flags** — для постепенного раскатывания фич
- **Self-hosted-first:** все зависимости должны иметь on-prem вариант (MinIO вместо S3, PostHog OSS, Sentry self-hosted и т.д.)
