# CALLCENTER.md — Колл-центр (телефония / SIP / Asterisk)

> Модуль колл-центра в админке SOS24: приём звонков 24/7, запись всех разговоров, создание заявок по звонку.
> **Статус:** проектирование (2026-06-24/25). Кода ещё нет. Читать вместе с `docs/QUESTIONS.md` раздел 14 и `docs/STAGE1.md`.

---

## 1. Цель и два типа звонков

Рабочее место оператора в админке, куда приходят звонки двух видов; все разговоры записываются; по звонку можно завести заявку.

1. **Внешний звонок** — клиент (или кто угодно) звонит на наш городской номер (PSTN) → SIP-транк → Asterisk → оператор.
2. **Звонок из приложения** — авторизованный пользователь звонит «в поддержку» из мобильного приложения (есть его `userId`). Голос идёт через WebRTC, без выхода в город.

## 2. Ключевое архитектурное решение

**Оба типа звонков сводим в один телефонный бэкбон на Asterisk** (а не делаем in-app отдельным самописным WebRTC-стеком).

Почему: иначе получаются два рабочих места оператора, две системы записи и два отчётных контура — двойная сложность. При единой схеме у оператора **один софтфон, одна очередь, одна запись, одна аналитика**.

Развенчанное заблуждение: «in-app звонок можно сделать просто через бэкенд без телефонии». Голос — это медиапоток (RTP/SRTP), а не HTTP; он не идёт «через бэкенд». Нужен WebRTC + TURN. Приложение делаем таким же WebRTC/SIP-клиентом к Asterisk, как и браузер оператора — тогда оба звонка приходят в одну очередь.

**Зафиксировано (2026-06-24):**
- Единый бэкбон на Asterisk. ✅
- Первая фаза — **внешние звонки** (инфраструктура уже есть), in-app — второй фазой. ✅

## 3. Целевая архитектура

```
[PSTN / внешний номер] ──SIP-транк──┐
                                    ▼
[Моб. приложение] ──WebRTC/SIP(WSS)──► Asterisk (PBX) ──► Очередь (ACD) ──► Оператор (браузер-софтфон SIP.js/WSS)
                                    │   • MixMonitor: запись          ▲ screen pop (карточка клиента)
                                    │   • события ARI/AMI             │
                                    ▼                                 │
                            NestJS (модуль call-center, ARI-app) ─Socket.IO─┘
                              • маршрутизация/решения (Stasis)   • CDR + метаданные звонка в Postgres
                              • выдаёт эфемерные SIP-креды        • перенос записи в MinIO
                              • данные для screen pop             • привязка звонка к SupportTicket
```

**Компоненты:**
- **Asterisk 20.18.2 под FreePBX 12.13** (`10.10.10.30`) — транк, очереди (app_queue/ACD), запись (MixMonitor), IVR, перевод. Очередь/запись/CDR не переизобретаем. ⚠️ FreePBX-managed: настройка через GUI + `*_custom.conf`.
- **Софтфон оператора** — браузерный, **SIP.js/JsSIP поверх WSS** (SIP over WebSocket) + DTLS-SRTP. Встроен в админку (раздел Call Center). Переиспользуем роль **SUPPORT** и наш Socket.IO (есть в модуле Поддержки).
- **Бэкенд NestJS, модуль `call-center`** — ARI-приложение (Stasis, REST+WebSocket к Asterisk): слушает события звонков, принимает решения по маршрутизации, шлёт события в админку через Socket.IO, выдаёт эфемерные SIP-креды, переносит записи в MinIO, пишет CDR/метаданные, связывает звонок с заявкой.
- **coturn (TURN)** — для WebRTC через NAT/мобильные сети. Обязателен (без него часть звонков не соединяется).

## 4. Два потока на одной схеме

**Внешний звонок:** PSTN → транк → Asterisk → очередь → свободный оператор. **Идентификация по CallerID**: номер матчим с `User.phone` (уникальный) → нашли → screen pop с карточкой клиента (профиль, полисы, история); не нашли → «неизвестный», оператор заводит данные.

**Звонок из приложения (Фаза 2):** приложение авторизовано (JWT, `userId`). Поднимает WebRTC/SIP-звонок в Asterisk и **передаёт `userId`** (кастомный SIP-заголовок `X-SOS24-User-Id` или эфемерная SIP-учётка, привязанная к userId). Asterisk кладёт в переменную канала → ARI-приложение шлёт в админку **точный screen pop** ещё до ответа.

В обоих случаях оператор в одном окне: принять / сбросить / удержание / перевод / DTMF / mute + кнопка **«Создать заявку»**.

## 5. Запись разговоров

- **MixMonitor** в Asterisk пишет каждый звонок (оба типа) → бэкенд по событию `recording finished` перекладывает файл в **MinIO**; метаданные (направление, номер, оператор, длительность, ключ записи, привязанная заявка) — в Postgres.
- **Юридическое (РУз):** объявление «разговор записывается» (IVR в начале) + контроль доступа к записям (RBAC: кто может слушать) + срок хранения. Заложить с самого начала (см. Q14.4).

## 6. Интеграция с заявками (омниканальность)

Расширяет модуль **Поддержки (M13)**: операторы уже SUPPORT-роль с рабочим местом и Socket.IO. Звонок создаёт/привязывается к `SupportTicket` (или новой сущности «обращение/лид»). В итоге чат и звонок по одному клиенту — в одной заявке.

## 7. Черновик модели данных (бэкенд)

> Предварительно, уточнить под реальную конфигурацию Asterisk.

`Call` (модель):
- `id`, `direction` (INBOUND_EXTERNAL | INBOUND_APP | OUTBOUND), `status` (RINGING | ANSWERED | MISSED | COMPLETED | FAILED)
- `externalNumber` (для внешних), `userId` (если идентифицирован/in-app), `operatorId` (User SUPPORT)
- `queue`, `startedAt`, `answeredAt`, `endedAt`, `durationSec`, `waitSec`
- `recordingKey` (MinIO), `asteriskChannelId`/`uniqueid` (корреляция с CDR/ARI)
- `ticketId` (привязка к SupportTicket), `note`
- индексы по `operatorId/startedAt`, `userId/startedAt`, `status`

## 8. Инфраструктура и сеть

**Где что стоит:**
- Asterisk: `10.10.10.30` (сеть `10.10.10.0/24`), Asterisk 20.18.2.
- Кластер SOS24: `10.10.38.0/24` (ноды) + nginx-ВМ `10.10.38.30`, MinIO `s3.sos24.uz`.

**⚠️ Связность (блокер прод-интеграции, Q14.1):** на 2026-06-25 сервера **не видят друг друга** — из пода кластера ARI(8088)/AMI(5038) Asterisk = timeout. Нужен **WireGuard site-to-site** между Asterisk и нодами кластера (маршрут с нод → `10.10.10.30`, обратный → `10.10.38.0/24`). В одну подсеть сводить **не обязательно**. Задача DevOps.

**Медиа (важно):** голос (RTP/SRTP) идёт оператор/приложение ↔ **Asterisk** (через TURN), **не через кластер**. Поэтому связность кластер↔Asterisk нужна только для control-плоскости (ARI/AMI) и выгрузки записей, а не для качества звонка.

**Порты:**
| Поток | Порт | Путь |
|---|---|---|
| Бэкенд → ARI | 8088 (или 8089 TLS) | приватно (WG-туннель) |
| Бэкенд → AMI | 5038 | приватно (WG-туннель) |
| Оператор/приложение → SIP over WSS | 8089 | публично (через наш nginx+SSL) |
| Медиа WebRTC | TURN + RTP (UDP, напр. 10000–20000) | публично (coturn/Asterisk) |
| SIP-транк провайдера | 5060/5061 | у Asterisk (на R5 5060 закрыт снаружи — уточнить, Q14.3) |
| Asterisk → MinIO | 443 | `s3.sos24.uz` (публично, проверить доступность) |

**Discovery (снято по SSH 2026-06-25):**
- **FreePBX 12.13 (Sangoma)** на **Asterisk 20.18.2**. ⚠️ Конфиг управляется FreePBX (GUI + БД) — правим только через `*_custom.conf`-инклюды и GUI, не редактируем managed-файлы руками.
- **ARI ВКЛЮЧЁН (2026-06-25).** Сделано через FreePBX-инклюды: `ari_general_custom.conf` → `enabled=yes`; ARI-юзер **`sos24cc`** (read-write) в `ari_additional_custom.conf`, пароль сгенерён на сервере, лежит в **`/root/.sos24_ari_cred`** (chmod 600) — забрать в наш secret при подключении бэкенда. ## 9. Безопасность

- **Эфемерные SIP-креды**: операторам и приложению бэкенд выдаёт короткоживущие учётки после логина (JWT) — не хардкодим.
- **Anti-toll-fraud**: запретить исходящие в город с операторских/клиентских учёток сверх нужного; ARI/AMI не выставлять в интернет (только через WG-туннель); fail2ban на Asterisk.
- **Доступ к записям** — по RBAC.
