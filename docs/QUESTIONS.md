### Q2.3 — Какие именно сервисы NAPP даёт страховой
**Статус:** ✅ ЗАКРЫТ (2026-06-04)
**Итог:** НАПП — полноценный агрегатор. Сам отдаёт: данные ТС (тех.паспорт → полные данные), ВУ водителя, ПИНФЛ → личные данные, КБМ коэффициент, кадастр недвижимости, реестр е-полисов. Подробно: `NAPP.md` раздел 3.

### Q2.4 — Поведение при недоступности NAPP
**Статус:** 🔴 OPEN
**Контекст:** НАПП API может быть недоступен. Варианты:
- (a) Блокировать оформление, показывать "Сервис временно недоступен, попробуйте позже"
- (b) Разрешить ручной ввод данных авто/водителя (без верификации через НАПП)
- (c) Ставить в очередь — НАПП регистрируется после восстановления
Для платёжных данных — вариант (a). Для данных авто — вариант (b) как fallback. **Нужно решение клиента.**

### Q2.5 — НАПП: production credentials
**Статус:** 🔴 OPEN (новый, 2026-06-04)

### Q2.6 — НАПП: `senderPinfl` страховой компании
**Статус:** 🔴 OPEN (новый, 2026-06-04)

### Q4.2 — Какой именно интерфейс Uzcard
**Статус:** 🔴 OPEN — критично
**Контекст:** Варианты:
- (a) **Uzcard Acquiring** (стандартный 3DS) — нужен PCI-DSS scope от страховой,
- (b) **Uzcard Tokenization / hosted fields** — данные карты идут напрямую в Uzcard, у нас только токен (PCI-DSS scope сильно меньше),
- (c) **через банк-партнёр** заказчика (свой расчётный счёт + acquiring).
Это критично для архитектуры платёжного модуля.
